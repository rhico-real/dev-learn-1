# RunHop Phase 2 Social Design

**Date:** 2026-04-14
**Status:** Proposed
**Type:** Phase Design & Module Breakdown

---

## Overview

Phase 2 extends RunHop from a race and organization platform into a social product. The goal is to add social content and interaction without breaking the modular monolith boundaries established in Phase 1.

This phase introduces four modules inside the Social context:

- `post`
- `reaction`
- `feed`
- `notification`

These modules should remain aligned with the existing architecture roadmap. The design deliberately avoids advanced distributed systems concerns for now. Phase 2 is meant to teach stronger service boundaries, richer read queries, cursor pagination, and side-effect handling while still staying manageable for a learner who just finished Phase 1.

---

## Goals

- Add a complete social layer on top of the existing Phase 1 platform
- Keep all new work inside the Social bounded context unless a small, explicit contract is needed from another context
- Preserve the modular monolith dependency rules
- Increase implementation difficulty slightly compared to Phase 1
- Teach more production-relevant patterns such as cursor pagination, read models, idempotent interactions, and event-driven side effects

## Non-Goals

- Real-time notifications
- Push notifications
- Recommendation algorithms
- Feed ranking and relevance scoring
- Media processing pipelines
- Background jobs and queues for social side effects
- Nested comment threads
- Reposts, shares, or quote-posts

---

## Why This Phase Exists

Phase 1 proved the project can handle identity, organizations, events, races, registrations, and follows. That gave RunHop a strong domain core. Phase 2 builds on those foundations:

- `follow` from Phase 1 becomes useful because it can now drive feed visibility
- `user`, `organization`, and `event` entities become social actors or social targets
- event participation becomes more engaging when content and notifications exist around it

This phase is where the project starts behaving like a platform instead of a back-office CRUD system.

---

## Recommended Implementation Order

The roadmap names the modules as posts, reactions, feed, and notifications. That order is also the correct implementation sequence.

### 1. Post Module

Build first because every other Phase 2 module depends on it.

### 2. Reaction Module

Build second because likes and comments depend on posts and introduce interaction rules such as uniqueness and side effects.

### 3. Feed Module

Build third because feed assembly depends on follows plus post visibility rules. This is the first read-heavy module in the system.

### 4. Notification Module

Build last because notifications should react to behavior from the first three modules and should not drive the core design.

## Social Context Boundaries

### Phase 2 Module Map

| Module | Responsibility |
|--------|----------------|
| `post` | Create, edit, delete, and list social posts |
| `reaction` | Likes and comments on posts |
| `feed` | Aggregated feed retrieval for a user |
| `notification` | In-app notifications generated by social actions |

### Cross-Context Rules

The Social context may depend on exported service contracts from other contexts, but it must not directly query repositories or Prisma models owned by those contexts.

Allowed examples:

- `PostService` asks `UserService.exists(userId)`
- `FeedService` asks `FollowService` or other exported contracts for followed targets
- `NotificationService` stores notifications about users, events, or organizations without taking ownership of those entities

Forbidden examples:

- `FeedService` directly querying `Organization` through Prisma
- `ReactionService` directly reading `User` table internals instead of going through identity contracts
- `NotificationService` embedding business logic from Event or Organization contexts

The boundary rule stays the same:

> A module may only consume another context's exported Service, never another context's Repository or Prisma model directly.

## Post Module Design

### Purpose

The `post` module is the core content module for Phase 2. It owns the canonical social content entity that later modules read or interact with.

### Scope

Phase 2 posts should stay intentionally simple:

- text-first content
- optional image URLs can be added later if your architecture already supports them cleanly
- no reposts
- no threaded discussions inside the post itself

### Ownership Model

To keep the first Phase 2 implementation teachable, the default authorship model should be:

- user-authored posts first

If later you want organization-authored or event-authored posts, those can be added as a controlled extension. Starting with user-authored posts reduces polymorphic complexity at the point where you are still learning the new social modules.

### Lifecycle Rules

- authenticated users can create posts
- authors can update their own posts
- authors can delete their own posts
- super admins may moderate if you want a platform-level override
- deleted posts should use soft delete if you want to preserve referential integrity and auditability

### Read Model

The post module should support:

- get one post
- list a user's posts
- list recent posts for feed assembly

The post entity should expose enough metadata for feed and notification use cases without making those modules responsible for content ownership.

## Reaction Module Design

### Purpose

The `reaction` module owns interaction on posts. In Phase 2, that means likes and comments.

### Likes

Likes should be modeled as a separate table with a database-level uniqueness rule:

- one user can like a given post only once

This teaches an important pattern from production systems:

- service-layer validation is not enough
- the database must enforce the invariant too

Recommended behavior:

- `like(postId)` creates the row if it does not exist
- `unlike(postId)` removes or deactivates the row
- repeated like attempts should be idempotent from the API consumer's perspective

### Comments

Comments should be a separate entity, not a variant inside the post table.

Phase 2 comments should support:

- create comment
- edit own comment
- delete own comment
- list comments for a post

Keep comments flat in Phase 2:

- no threaded replies
- no mention parsing
- no moderation workflows beyond simple ownership checks

### Side Effects

Reaction events are a natural source of notifications:

- user liked your post
- user commented on your post

The reaction module should emit domain events rather than directly creating notifications through hard service coupling.

---

## Feed Module Design

### Purpose

The `feed` module is a read-oriented module that composes data from posts and follow relationships into a user-facing stream.

This is the main learning upgrade in Phase 2. It is less about CRUD and more about query design.

### Feed Inputs

The feed should primarily be derived from:

- entities the current user follows from Phase 1
- post visibility rules
- reverse chronological ordering for Phase 2

This should stay simple and deterministic:

- no recommendation scoring
- no trending algorithm
- no mixed ranking signals

### Pagination Strategy

Feed queries should use cursor pagination, not offset pagination.

Reasoning:

- offset pagination becomes unstable as rows are inserted
- cursor pagination is a more realistic production pattern
- this phase should push your learning slightly beyond basic list endpoints

### Query Discipline

The feed module should force careful thinking about:

- sorting by stable fields such as `createdAt` and `id`
- joining or composing enough data for the client response
- avoiding sloppy N+1 reads
- returning consistent pagination metadata

### Response Shape

Feed items should contain:

- post data
- author summary
- reaction counts summary
- viewer-specific flags when appropriate, such as whether the current user liked the post

This creates a useful read model without forcing the client to make multiple follow-up requests for every visible post.

---

## Notification Module Design

### Purpose

The `notification` module owns in-app notifications generated by system activity.

In Phase 2, notifications should remain intentionally narrow:

- persisted in the database
- listed in-app through an API
- mark-as-read support
- no push delivery
- no email delivery

### Example Triggers

- someone liked your post
- someone commented on your post
- future expansion: event-related or follow-related notifications

### Delivery Model

Notifications should be created synchronously in Phase 2 through event listeners inside the modular monolith.

Use NestJS event emission patterns so the publisher does not depend on the notification service directly:

```text
ReactionService.createLike()
  -> emits 'social.post.liked'
NotificationListener
  -> creates notification row
```

This preserves clean boundaries now and leaves room for queue-backed processing later in Phase 4.

### Read Rules

The notification module should support:

- list current user's notifications
- mark one notification as read
- mark all notifications as read
- unread count

## Data Modeling Guidance

### Core Phase 2 Entities

| Entity | Notes |
|--------|-------|
| `Post` | Social content owned by an author |
| `PostLike` | Unique user-post relationship |
| `PostComment` | Flat comment under a post |
| `Notification` | User-targeted in-app notification |

### Suggested Data Rules

- `Post`
  - has `id`, `authorId`, `content`, timestamps, optional soft-delete field
- `PostLike`
  - unique on `(postId, userId)`
- `PostComment`
  - belongs to `postId` and `authorId`
- `Notification`
  - belongs to a recipient user
  - has a type enum and enough foreign key reference data to render safely

### Counts and Aggregates

Phase 2 should prefer correctness and clarity over premature optimization.

Good enough for now:

- query counts directly
- include counts in response DTOs
- denormalize only later if needed

Do not introduce cached counters or feed fan-out tables yet unless a measured performance problem forces it.

## API Design Direction

All endpoints remain under `/api/v1`.

### Post Endpoints

- `POST /posts`
- `GET /posts/:id`
- `GET /users/:userId/posts`
- `PATCH /posts/:id`
- `DELETE /posts/:id`

### Reaction Endpoints

- `POST /posts/:id/likes`
- `DELETE /posts/:id/likes`
- `GET /posts/:id/comments`
- `POST /posts/:id/comments`
- `PATCH /comments/:id`
- `DELETE /comments/:id`

### Feed Endpoints

- `GET /feed`

### Notification Endpoints

- `GET /notifications`
- `PATCH /notifications/:id/read`
- `PATCH /notifications/read-all`

Exact DTO fields and response shapes belong in the implementation plan. This design only fixes the module responsibilities and behavior constraints.

## Permissions and Ownership Rules

### Baseline Rules

- only authenticated users can create posts, likes, comments, and read personal feeds
- users can edit or delete only their own posts and comments
- users can only read or mark their own notifications
- super admins may receive moderation overrides if the platform needs them

### Visibility Rules

Keep Phase 2 visibility simple. A post is visible if:

- it is not deleted
- the viewer is allowed to see the author's content under current business rules

Do not add privacy matrices in this phase unless you already decided them elsewhere. The feed should be hard enough because of query composition, not because of speculative privacy complexity.

---

## Testing Strategy

Phase 2 should continue the Phase 1 discipline:

- service-level unit tests for business rules
- controller tests where behavior mapping matters
- e2e coverage for critical flows

The implementation plan should push harder on the following test categories:

- uniqueness and idempotency for likes
- ownership checks for post and comment edits
- cursor pagination correctness
- notification creation from emitted social events
- feed ordering and visibility filtering

This is the right phase to become more comfortable writing tests for read-heavy logic instead of only mutation flows.

---

## Learning Objectives

Compared with Phase 1, Phase 2 should stretch you in these areas:

- modeling one-to-many and many-to-many social interactions cleanly
- distinguishing canonical write models from read models
- designing cursor pagination
- preventing N+1 query patterns
- keeping side effects decoupled through domain events
- knowing when not to optimize too early

This is the intended challenge increase: not overwhelming, but clearly beyond beginner CRUD.

---

## Risks and Guardrails

### Risk 1: Overcomplicating authorship

If posts support users, organizations, and events immediately, the model becomes polymorphic too early.

Guardrail:

- start with user-authored posts unless you have a strong immediate requirement otherwise

### Risk 2: Turning feed into an algorithm project

A social feed can spiral into ranking, recommendation, and caching work.

Guardrail:

- Phase 2 feed is chronological and follow-driven only

### Risk 3: Hard-coupling notifications to write flows

Direct service calls from every module into notification logic make the Social context harder to evolve.

Guardrail:

- publish domain events and let listeners create notifications

### Risk 4: Premature performance work

Denormalized counters, fan-out tables, and caching can hide core design mistakes if introduced too early.

Guardrail:

- start with clear relational models and straightforward queries

---

## Docs and Learning Experience

The Phase 2 implementation plan should preserve the strengths of the Phase 1 plan:

- explicit file paths
- step-by-step task breakdown
- concrete test guidance
- explanation of why each module exists

But it should raise the difficulty slightly:

- fewer reminders about NestJS basics you already practiced
- stronger emphasis on query design and service boundaries
- more deliberate testing of read models and side effects

The intended tone is:

- still guided
- still teachable
- slightly more demanding

RunHop now has a cleaner phase progression:

| Phase | Focus |
|-------|-------|
| **Phase 1** | Platform core: auth, users, organizations, events, races, registrations, follows |
| **Phase 2** | Social layer: posts, reactions, feed, notifications |

Phase 2 should make the platform feel alive. It is the bridge between a functional MVP and a product with user engagement loops.
