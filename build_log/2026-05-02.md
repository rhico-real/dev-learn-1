# Build Log: May 2, 2026
**Project:** RunHop — a social + event platform for running races
**Stack:** NestJS · TypeScript · PostgreSQL · Prisma · Redis · Docker · Docker Compose · Colima
**Tasks:** Phase 4 Docker packaging pass, Compose debugging, Prisma runtime fixes, docs gotchas capture

## What I Did

Today was a full Docker reality check for Phase 4. The code path was mostly there, but the container path was not. I worked through the actual build and startup sequence end to end instead of assuming the Dockerfile and Compose file were “close enough.”

- Built the app image locally and fixed the first hard failure in `Dockerfile`: `npm run builder` was wrong because `package.json` only defines `build`.
- Traced Prisma generation failures back to the ERD generator in `prisma/schema.prisma`. Commented out the `generator erd` block so Docker only has to generate the Prisma client, not render Mermaid diagrams through Puppeteer.
- Moved the image base toward Debian-based Node images after Alpine started surfacing browser/tooling compatibility issues around Prisma ERD generation and system libraries.
- Verified the actual Nest output path with `npm run build` and `find dist -maxdepth 3 -type f`. The real runtime entrypoint is `dist/src/main.js`, not `dist/main.js`.
- Worked through Prisma runtime issues in the final image:
  - missing generated client/runtime mismatch around `Platform`
  - database tables not existing because migrations had not run
  - connection failures because the app container was still using `localhost` for Postgres and Redis
- Identified the split between local env values and container env values:
  - local app mode needs `localhost`
  - full Compose app mode needs `postgres` and `redis`
- Brought the full stack up successfully under Compose with:
  - `dev-learn-1-app-1`
  - `dev-learn-1-postgres-1`
  - `dev-learn-1-redis-1`
- Captured the day’s Docker and Colima gotchas in a new docs guide:
  - `docs/guides/docker-gotchas.md`
  - linked from `docs/_sidebar.md`
  - linked from `docs/README.md`
- Added a Colima note into the Phase 4 plan so the rendered HTML docs also explain the local runtime setup.

## Struggles

**The biggest trap was assuming build success meant runtime correctness.** Docker happily produced an image even while the app still had a bad entrypoint, stale Prisma runtime assumptions, and wrong container networking values. The real debugging only started after checking `docker ps -a` and `docker logs dev-learn-1-app-1`.

**`localhost` inside Compose was the most expensive configuration mistake.** It worked fine for running Nest directly on the host, so it looked correct at first glance. Inside the `app` container, though, `localhost` meant “this same container,” which caused both `ECONNREFUSED 127.0.0.1:6379` and Prisma `P1001` against `localhost:5432`.

**Prisma generation is easy to get wrong in multi-stage builds.** Running `npx prisma generate` in the builder stage felt like enough, but the runner stage installs a fresh `node_modules`. That meant the final image could still boot with a client/runtime mismatch unless generation or generated artifacts were handled for the actual runtime image.

**Port conflicts were messier than just “Docker is broken.”** The real blockers were leftover containers, Homebrew services, and an SSH tunnel all competing for `5432` and `6379`. `docker compose down` was not enough because it only tears down the current project stack.

## STAR Story (Apple Behavioral)

**Situation:** Phase 4 already had a Dockerfile, a Compose file, Redis integration, Prisma, and push notification changes on paper, but that did not mean the packaged application actually ran. The app image built, but the container path kept failing at different layers: bad script name, ERD generation side effects, wrong runtime entrypoint, stale Prisma runtime behavior, and container networking mistakes.

**Task:** Get the full local containerized stack into a debuggable, repeatable state and turn the one-off terminal lessons into documentation the next session could reuse.

**Action:** I worked the failures in runtime order instead of jumping around. First I fixed the obvious Dockerfile script issue. Then I isolated Prisma ERD generation as a build-time concern and removed it from the container path. After that I verified the real Nest build output instead of assuming `dist/main.js`. Once the app container could start far enough to log, I used `docker ps -a` and `docker logs` to separate entrypoint problems, Prisma client/runtime mismatches, missing migrations, and wrong env values. In parallel, I captured the recurring failure modes into a dedicated docs guide and linked it into the docs navigation so the rendered HTML docs would surface it too.

**Result:** The stack now boots far enough to behave like a real Compose app instead of a theoretical one, and the repeated Docker/Colima/Prisma mistakes from this session are documented in one place. The main gain was not just a running container path; it was converting a messy debugging session into a reusable operational guide for the project.

I spent today turning RunHop's Docker setup from "there is a Dockerfile in the repo" into "the stack actually runs."

That distinction matters more than it sounds.

RunHop is a NestJS backend I'm building in public for a social + event platform for races. By this point the project already had a `Dockerfile`, a `docker-compose.yml`, Redis, Prisma, and enough infrastructure to look production-minded on paper. But the first real end-to-end build exposed the usual truth: packaging is where assumptions get audited.

## The First Failure Was Boring, Which Was Good

The first image build failed on this:

```dockerfile
RUN npm run builder
```

The app didn't have a `builder` script. It had:

```json
"scripts": {
  "build": "nest build"
}
```

So the first fix was simple: use `npm run build`.

That was a useful reminder that some Docker failures are not "Docker problems" at all. They're just the containerized version of a command you'd immediately notice on the host.

## Prisma ERD Generation Broke The Build Path

The next failure was more interesting.

Prisma generation was failing because the schema included an ERD generator:

```prisma
generator erd {
  provider = "prisma-erd-generator"
  output   = "../docs/erd.svg"
}
```

That pulled Mermaid/Puppeteer into the build path and triggered browser/runtime issues inside the container. The container did not need an ERD image to run the app. It only needed the Prisma client.

So I commented out the ERD generator for the container flow and kept:

```prisma
generator client {
  provider = "prisma-client-js"
}
```

This was one of those "separate build-time documentation tooling from runtime application packaging" moments. The container should not be responsible for generating docs artifacts if they aren't needed to serve traffic.

## The Entry Point Was Wrong Even After A Successful Build

Once the image built, the app container still crashed:

```text
Error: Cannot find module '/app/dist/main.js'
```

That looked like a copy problem at first. It wasn't. The real issue was a bad assumption about Nest output.

I checked the actual build result:

```bash
npm run build
find dist -maxdepth 3 -type f
```

And the real entry file was:

```text
dist/src/main.js
```

So the runtime command had to match that:

```dockerfile
CMD ["node", "dist/src/main.js"]
```

That was a good reminder to inspect `dist/` instead of guessing what the framework "usually" emits.

## Builder Stage Prisma Generation Was Not Enough

I also hit a Prisma runtime mismatch around an enum imported from `@prisma/client`:

```text
TypeError: Cannot convert undefined or null to object
```

The project had added a `Platform` enum for device tokens, and the app crashed during DTO decorator evaluation.

The root issue was that multi-stage builds create separate environments. Generating Prisma client in the builder stage doesn't automatically make the runner stage correct, because the runner installs its own fresh `node_modules`.

That pushed me toward a more explicit rule: the final runtime image must either generate Prisma client itself or receive the generated runtime artifacts intentionally.

## `localhost` Inside Compose Was The Most Expensive Config Mistake

The app later failed with:

```text
connect ECONNREFUSED 127.0.0.1:6379
PrismaClientInitializationError: Can't reach database server at `localhost:5432`
```

This one is worth writing down because it's so common.

My local `.env` was correct for running Nest directly on my Mac:

```env
DATABASE_URL=postgresql://runhop:runhop@localhost:5432/runhop
REDIS_HOST=localhost
REDIS_PORT=6379
```

But inside the Compose `app` container, `localhost` means the app container itself. It does not mean the Postgres or Redis service containers.

For full Compose mode, the app must use service names:

```yaml
environment:
  DATABASE_URL: postgresql://runhop:runhop@postgres:5432/runhop
  REDIS_HOST: redis
  REDIS_PORT: 6379
```

That was the cleanest example of why local-host and container-host config should not be treated as interchangeable.

## Port Conflicts Were Not Compose's Fault

I also kept hitting:

```text
Bind for :::5432 failed: port is already allocated
Bind for :::6379 failed: port is already allocated
```

The actual causes were:

- old containers still running
- Homebrew Postgres and Redis services
- an SSH tunnel listening on the same ports

`docker compose down` was not enough because it only tears down the current Compose stack. It does not stop everything else on the host.

The reliable debug set became:

```bash
docker ps
docker ps -a
lsof -i :5432
lsof -i :6379
brew services list
docker logs <container-name>
```

Once I accepted that port conflicts are a host-level problem, not just a Compose problem, the debugging got much faster.

## Colima Was Fine, But It Added One More Layer To Understand

I'm using Colima instead of Docker Desktop on macOS. That part was mostly fine, but it matters conceptually:

- `docker` is the CLI
- Colima is the runtime/daemon provider on the machine

At one point the Colima profile showed as broken even though the socket path still existed, so `docker ps` failed until the runtime itself was healthy again.

That wasn't a reason to abandon Colima. It was just another reminder that local container tooling has layers, and it's worth knowing which one you're debugging.

## What I Documented From The Session

By the end of the day I wrote a dedicated `Docker Gotchas` guide into the project docs. I wanted one place to capture the non-obvious lessons:

- `docker stop` vs `docker compose down`
- `up` vs `start`
- why `--build` is not always needed
- why `docker ps` and `docker ps -a` tell different stories
- the `localhost` trap inside containers
- why build success is not runtime success
- where Docker images actually "live" under Colima

The actual technical fixes mattered, but documenting the reasoning mattered almost as much. Otherwise the same mistakes come back the next time the stack is touched.

## Takeaway

The useful lesson from today wasn't "I got Docker working." It was that Docker problems get much easier once you stop treating them as one big category.

There are at least four separate layers:

1. build commands
2. runtime entrypoint
3. container networking
4. application startup dependencies

If you debug them in order, the stack gets understandable fast. If you mix them together, every error looks like infrastructure chaos.
