## Struggles

**`update` vs `updateMany` for `markAllAsRead`.** First pass used `prisma.notification.update()` with `where: { recipientId: userId }`. Prisma rejected it because `update` requires a unique field in `where`, and `recipientId` isn't unique — one user has many notifications. The fix was `updateMany`, which accepts non-unique filters and returns `{ count: number }` instead of a record.

**Tried composite index syntax inside `updateMany`.** Wrote `where: { recipientId_readAt: { recipientId: userId, readAt: null } }` thinking I needed to reference the `@@index` explicitly. That syntax is for `@@unique` composite lookups in `findUnique`, not for `updateMany`. The correct shape is just flat fields: `where: { recipientId: userId, readAt: null }`. Postgres uses the index automatically.

**E2e flakiness from shared test emails.** `reactions.e2e-spec.ts` and `post.e2e-spec.ts` both registered `post-e2e-test@test.com`. Whichever file ran first got the user; the other got a conflict error and every downstream test failed. Fix: unique emails per file.

**Event listener placement.** Initial instinct was to put the event emit functions in the notification module. That would have re-coupled `ReactionService` to `NotificationService` — the exact thing events are supposed to prevent. The clean split: constants in a shared file, `ReactionService` emits, `NotificationService` listens via `@OnEvent`.

- The event emitter pattern decoupled reactions from notifications cleanly. `ReactionService` fires and forgets, `NotificationService` independently handles the side effect.
- `readAt: null` filter on `markAllAsRead` means Postgres only touches unread rows, not every notification. The `@@index([recipientId, readAt])` was already in the schema waiting for exactly this query.
- The notification e2e test is solid: 10 actors generating 40 notifications, then pagination, cursor dedup, mark-one, mark-all, and final unread count = 0. Gives high confidence without mocking anything.
- Three modules shipped in one session. The patterns from posts and reactions (cursor pagination, ownership checks, Prisma error mapping) compounded — each module was faster to build than the last.

## STAR Story (Apple Behavioral)

**Situation:** I was building the notification service and needed `markAllAsRead` to bulk-update every unread notification for a user. The Notification table has an `@@index([recipientId, readAt])` designed for this query.

**Task:** Write a bulk update that marks all unread notifications as read efficiently, using the existing index.

**Action:** First attempt used `prisma.notification.update()` with `where: { recipientId: userId }`, which Prisma rejected — `update` requires a unique field. Switched to `updateMany`. Then I tried referencing the index explicitly with composite syntax (`recipientId_readAt: { ... }`), which is only valid for `@@unique` in `findUnique`. Stripped it back to flat fields: `where: { recipientId: userId, readAt: null }`. The `readAt: null` filter scopes the write to only unread rows, and Postgres picks up the composite index automatically.

**Result:** `markAllAsRead` hits only the rows that need updating, uses the index the schema already provides, and the e2e test confirms the full cycle: 40 unread → mark all → unread count returns 0.

I'm building RunHop in public, a social + event platform for running races. Today I shipped three modules: comments, feed, and notifications.

The interesting part wasn't the CRUD — it was the notification wiring.

## The Problem

When a user likes or comments on a post, the post owner should get a notification. The obvious approach:

```ts
// inside ReactionService.like()
await this.notificationService.createLikeNotification(userId, postId);
```

That works. But now `ReactionService` depends directly on `NotificationService`. The reaction module imports the notification module. Two bounded contexts are now coupled.

## The Event-Driven Approach

NestJS has `@nestjs/event-emitter` built on `eventemitter2`. The pattern:

1. Define event name constants in a shared file:

```ts
// notification-events.ts
export const NotificationEventTypes = {
    POST_LIKE: 'social.post.liked',
    POST_COMMENT: 'social.post.comment',
};
```

2. `ReactionService` emits after a successful like:

```ts
this.eventEmitter.emit(NotificationEventTypes.POST_LIKE, {
    actorId: userId,
    postId,
});
```

3. `NotificationService` listens:

```ts
@OnEvent(NotificationEventTypes.POST_LIKE)
async createLikeNotification(dto: CreateNotificationDto) {
    const post = await this.prisma.post.findUnique({
        where: { id: dto.postId },
        select: { authorId: true },
    });

    if (!post) throw new NotFoundException('Post not found');
    if (post.authorId === dto.actorId) return null;

    return this.prisma.notification.create({
        data: {
            recipientId: post.authorId,
            actorId: dto.actorId,
            postId: dto.postId,
            type: NotificationType.POST_LIKED,
        },
    });
}
```

`ReactionService` doesn't know notifications exist. It fires and forgets.

## The Recipient Isn't in the Payload

One thing that tripped me up: `createLikeNotification` takes `actorId` and `postId` — but not `recipientId`. The recipient is the post owner, so the notification service resolves it by loading the post first.

There's also a business rule: if `actorId === post.authorId`, skip the notification. You don't notify someone that they liked their own post.

## A Prisma Gotcha: update vs updateMany

For `markAllAsRead`, I needed to bulk-update every unread notification for a user. First attempt:

```ts
await this.prisma.notification.update({
    where: { recipientId: userId },
    data: { readAt: new Date() },
});
```

Prisma rejected this. `update` requires a unique field in `where` — and `recipientId` is not unique. The fix is `updateMany`:

```ts
await this.prisma.notification.updateMany({
    where: { recipientId: userId, readAt: null },
    data: { readAt: new Date() },
});
```

Two details matter here:

1. `readAt: null` scopes the write to unread rows only — no wasted I/O on already-read notifications
2. The schema has `@@index([recipientId, readAt])` — Postgres uses it automatically, no need to reference it in the query

## Testing the Full Chain

Unit tests cover each side independently:
- `reaction.service.spec.ts`: assert `mockEventEmitter.emit` was called with the right event and payload
- `notification.service.spec.ts`: call `createLikeNotification` directly, assert `prisma.notification.create` got the right data

The event wiring itself (that `@OnEvent` actually connects to the handler) gets covered in e2e. The test creates 10 actors who each like and comment 3 times on a post, then asserts:
- `GET /notifications` returns 20 (default page size)
- `GET /notifications?limit=5` returns 5
- Second page with cursor has no duplicates
- Unread count = 40
- Mark one as read → `readAt` is set
- Mark all as read → unread count = 0

## Takeaway

Events are one of those patterns that sound like overengineering until you see what the alternative looks like. The alternative here was a direct import chain across bounded contexts. The event approach is the same amount of code, and it keeps each module's dependency graph clean.
