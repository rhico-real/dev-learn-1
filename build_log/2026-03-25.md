I'm building RunHop in public. This is day 8.

Today I built the Organization domain: organizations, membership, and role management. The implementation was straightforward — but there was one design decision worth talking about.

## Should Membership Logic Live in the Org Service?

It's tempting. `OrgService.addMember()`. Clean, simple, one fewer module.

But `OrgMembership` isn't just a join table. It has its own behavior: adding members, removing members, updating roles (OWNER, ADMIN, MEMBER). That's a distinct set of operations with its own validation rules and its own controller endpoints.

If I put all of that in `OrganizationService`, I'd end up with a service that does two things: manage organizations *and* manage membership. That's the kind of coupling that feels fine at 200 lines and becomes painful at 800.

So I split it:

```typescript
// organization-context.module.ts
@Module({
  imports: [OrganizationModule, OrgMembershipModule],
})
export class OrganizationContextModule {}
```

`OrganizationModule` handles CRUD on the org itself. `OrgMembershipModule` handles everything about who's in the org and what role they have. Both live under the same bounded context (`organization/`), but they're separate modules with separate services and controllers.

## The Nested Route Challenge

Membership endpoints are nested: `/organizations/:orgId/members`. The membership controller needs `orgId` from the URL, but it lives in a different module than the org controller.

In NestJS, this works fine — the `@Param('orgId')` decorator pulls from the route regardless of which module the controller belongs to. But you have to set the controller's route prefix correctly:

```typescript
@Controller('organizations/:orgId/members')
export class OrgMembershipController {
  @Post()
  addMember(
    @Param('orgId') orgId: string,
    @Body() dto: AddMemberDto,
  ) {
    return this.membershipService.addMember(orgId, dto);
  }

  @Patch(':memberId/role')
  updateRole(
    @Param('orgId') orgId: string,
    @Param('memberId') memberId: string,
    @Body() dto: UpdateRoleDto,
  ) {
    return this.membershipService.updateRole(orgId, memberId, dto);
  }
}
```

Not complicated once you see it, but if you've never split nested routes across modules before, it's not obvious that this just works.

## The Payoff of Week 1 Architecture

The entire org domain — 14 files, 553 lines — took one session. No decisions about folder structure, no rewiring of modules, no wondering where the DTOs go. The bounded-context pattern from week 1 answered every question before I asked it.

That's the return on investing in architecture early. It's not that it makes the first feature faster — it makes every feature after the first one faster.
