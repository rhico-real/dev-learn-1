## Struggles

**Route collision between `FollowController` and `UserController`.** The `GET /users/following` route was being swallowed by `UserController`'s `GET /users/:id` тАФ NestJS treated `"following"` as the `:id` param and threw a 404 from `UserService.findById("following")`. The fix was changing the route to `GET /users/me/following`. Parameterized routes match any string, so static segments on the same path depth need to go first or use a different pattern. The `users/:id/followers` route was fine because it has three segments vs two.

**`TransformInterceptor` eating boolean `false`.** `isFollowing` returns `!!follow`, which can be `false`. The interceptor had `if(!response) return { data: null }` тАФ and `!false` is `true`, so it was converting a legitimate `false` return into `null`. Took a bit to trace because the service logic was correct but the response body showed `{ data: null }`. Fixed by checking `=== null || === undefined` instead of using falsy coercion.

**Jest e2e test timing.** The `beforeAll` that creates multiple test accounts in a loop needed a timeout bump to 30000ms тАФ 5 sequential HTTP calls (register + follow each) exceeded Jest's default 5000ms. Also had to use `--runInBand` in the e2e script because parallel test files were racing on Redis connections. One file's `app.close()` could tear down the Redis client while another file's setup was still running.

- The polymorphic pattern worked clean. One table, one service, one set of endpoints тАФ no duplication across USER/ORG/EVENT follow logic.

## STAR Story (Apple Behavioral)

**Situation:** The Follow module needed an `isFollowing` endpoint that returns a boolean тАФ `true` if the user follows a target, `false` if not. The service logic was correct (`!!follow`), but the API was returning `{ data: null }` instead of `{ data: false }` for the negative case.

**Task:** Trace why a correct service return value was being transformed into an incorrect API response, and fix it without breaking existing endpoints that legitimately return `null`.

**Action:** I traced the response pipeline from service тЖТ controller тЖТ interceptor. The `TransformInterceptor` had a guard: `if (!response) return { data: null }`. JavaScript's falsy coercion meant `!false === true`, so the interceptor was catching `false` as if it were an empty response. I changed the guard to `if (response === null || response === undefined)` тАФ strict equality checks that distinguish "no data" from "the data is the boolean false." Verified with unit tests on the interceptor and e2e tests on the `isFollowing` endpoint for both true and false cases.

**Result:** The `isFollowing` endpoint correctly returns `{ data: true }` or `{ data: false }`. All other endpoints that return objects or `null` continue to work as before. The fix is a one-line change with no side effects, but it exposed a class of bugs where interceptors/middleware silently transform valid falsy values.

## The Module: Polymorphic Follows

Users can follow three types of things: other users, organizations, and events. Instead of three separate tables (`user_follows`, `org_follows`, `event_follows`), there's one `Follow` table with a `targetId` and `targetType` enum:

```typescript
const targetTypeMap: Record<string, TargetType> = {
    USER: TargetType.USER,
    ORGANIZATION: TargetType.ORGANIZATION,
    EVENT: TargetType.EVENT,
};
```

The trade-off: the database can't enforce foreign keys on `targetId` because it could point to any of three tables. So validation happens at the application layer:

```typescript
private async validateTargetExists(targetId: string, targetType: TargetType): Promise<void> {
    let exists: boolean;

    switch (targetType) {
        case TargetType.USER:
            exists = await this.userService.exists(targetId);
            break;
        case TargetType.ORGANIZATION:
            exists = await this.orgService.exists(targetId);
            break;
        case TargetType.EVENT:
            exists = await this.eventService.exists(targetId);
            break;
        default:
            throw new BadRequestException(`Invalid target type: ${targetType}`);
    }

    if (!exists) {
        throw new NotFoundException(`${targetType} not found.`);
    }
}
```

This makes the Follow module the only Phase 1 module that imports all three other contexts. It validates across boundaries but doesn't understand the internals of any of them тАФ it only calls `.exists()`.

## The Bug: `!false === true`

The `isFollowing` method checks whether a follow relationship exists and returns a boolean:

```typescript
async isFollowing(followerId: string, targetId: string, targetType: string) {
    const follow = await this.prismaService.follow.findUnique({
        where: {
            followerId_targetId_targetType: {
                followerId, targetId,
                targetType: targetTypeMap[targetType],
            },
        },
    });

    return !!follow;
}
```

This returns `true` or `false`. Clean. Correct. But the API was returning `{ data: null }` instead of `{ data: false }`.

The culprit was the global `TransformInterceptor`:

```typescript
intercept(context: ExecutionContext, next: CallHandler): Observable<any> {
    return next.handle().pipe(
        map((response) => {
            if (!response) {          // <-- this line
                return { data: null };
            }
            if (response.data && response.meta) {
                return response;
            }
            return { data: response };
        }),
    );
}
```

`!false` evaluates to `true`. So when `isFollowing` returned `false`, the interceptor treated it as an empty response and replaced it with `null`.

The fix:

```typescript
if (response === null || response === undefined) {
    return { data: null };
}
```

Strict equality. `false` passes through to `return { data: response }` and becomes `{ data: false }`.

## Route Collisions in NestJS

Another gotcha: I had `GET /users/following` in the FollowController and `GET /users/:id` in the UserController. NestJS matched `:id` first, interpreting "following" as a user ID, and returned a 404 from `UserService.findById("following")`.

Parameterized routes are greedy тАФ `:id` matches any string. If you have a static route at the same depth, the param route wins if it's registered first. The fix was changing to `GET /users/me/following`, which has a different shape and doesn't collide.

Worth knowing: `GET /users/:id/followers` (three segments) doesn't conflict with `GET /users/:id` (two segments). NestJS only matches routes with the same segment count.

## Takeaway

Falsy coercion in middleware or interceptors is a subtle bug class. `!response` catches `false`, `0`, and `""` along with `null` and `undefined`. If your API can legitimately return any falsy value, use strict equality checks. The interceptor worked fine for months because every endpoint returned objects or null тАФ it only broke when `isFollowing` introduced the first boolean return.
